# 用 Codex 开发 HR Excel 工具的完整方法

这份文档面向业务负责人、项目协调人和非专职开发人员。目标是：拿到 HR、财务、行政等部门给出的 Excel 自动化需求后，能够把需求、数据源、模板和验收标准描述清楚，让 Codex 这类代码 Agent 完成分析、实现、验证、修改，并最终交付成可使用的工具。

本文档以 `HR工具箱` 这个项目为例，但方法适用于其他 Excel 自动化工具。

## 一、这个工具的技术架构

当前 HR 工具箱采用的是本地桌面工具架构：

```text
hr-toolkit/
  hr_toolkit/
    gui.py                 # 桌面界面，给 HR 点击使用
    tools/                 # 每个需求一个工具模块
    common/                # Excel、文件、日期等公共能力
    templates/             # 内置干净模板
  tests/                   # 自动化测试
  scripts/                 # 打包、发布、版本更新脚本
  release/                 # 给服务器更新用的发布文件
```

### 为什么用这个架构

- 业务使用环境以 Windows 为主，且存在旧系统环境，桌面 exe 是交付成本较低的方案。
- Excel 文件需要保留格式、公式、边框、合并单元格，所以核心处理用 `openpyxl`，不优先用 `pandas`。
- 每个需求拆成一个独立工具模块，降低后续新增需求时的代码耦合。
- GUI 只负责选择文件、显示日志、调用工具，业务逻辑放在 `tools/` 里，便于测试。
- 打包后业务使用者只需要打开 exe，不需要安装 Python。
- 更新包放在服务器静态目录，程序启动时检查更新。

### 一个需求对应的典型代码结构

```text
hr_toolkit/tools/social_security_summary.py     # 需求1：社保汇总
hr_toolkit/tools/attendance_summary.py          # 需求2：考勤统计
hr_toolkit/tools/insurance_ledger.py            # 需求3：保险台账
hr_toolkit/tools/salary_split.py                # 需求4：工资拆分
hr_toolkit/tools/salary_merge.py                # 需求5：工资合并
hr_toolkit/tools/personnel_change_merge.py      # 需求6：异动汇总
hr_toolkit/tools/archive_import.py              # 需求7：档案入库
hr_toolkit/tools/folder_rename.py               # 需求8：文件夹改名
```

原则：一个需求一个模块。不要把多个业务需求写在一个大文件里。

## 二、给 Codex 的输入必须包含什么

要让 Codex 准确实现，不能只说“帮我做一个工具”。至少要给它 5 类信息。

### 1. 需求表

通常是 HR 给的“问题汇总表”。

需要说明：

- 哪一行是本次要做的需求。
- 需求编号是什么，例如需求1、需求2。
- HR 原始描述是什么。
- 备注里有没有补充说明。

建议说法：

```text
请先看《问题汇总表》中的需求1，结合备注分析该需求的业务目标。
先不要写代码，先告诉我：
1. 数据源是否够；
2. 模板是否够；
3. 哪些能直接实现；
4. 哪些需要问 HR 确认。
```

### 2. 数据源

数据源就是 HR 实际会上传的 Excel、文件夹、压缩包。

必须告诉 Codex：

- HR 实际上传的是单个文件、多个文件、文件夹，还是 zip 压缩包。
- 文件名是否有业务含义，例如 `唐人四川社保账单.xlsx`。
- 文件夹名是否有业务含义，例如 `唐人四川`。
- 是否存在 `.xls` 旧格式。
- 是否会有多个 sheet。
- 哪些 sheet 要处理，哪些 sheet 不用处理。

建议说法：

```text
这是 HR 真实会上传的数据源，不要自己造业务字段。
请优先兼容这些真实文件。
如果测试需要构造样例数据，必须说明哪些是构造数据，哪些来自真实模板。
```

### 3. 输出模板

模板比需求文字更重要。Excel 工具最终能不能用，主要看输出是不是和模板一致。

必须告诉 Codex：

- 哪个文件是最终输出模板。
- 哪些行是标题、表头、说明文字。
- 哪些单元格有公式。
- 哪些地方要保留边框、字体、合并单元格。
- 新增行是否要复制上一行格式。
- 输出时是否允许改变背景色。

建议说法：

```text
这个文件是最终输出模板。
输出结果必须尽量保持模板格式：
- 标题、表头、边框、列宽、冻结窗格要保留；
- 新增行要有边框；
- 日期格式要和模板一致；
- 不要额外生成业务使用者不需要关注的隐藏文件或 manifest 文件。
```

### 4. HR 确认结果

需求不清楚时，不要让 Codex 自行假设，要让 Codex 帮你整理“需要业务确认的问题”。

建议说法：

```text
把需要确认的问题改成业务确认方可以直接判断的问题。
避免使用代码、数据结构等技术术语。
每个问题都要能让业务确认方直接回答“是/否”或者提供一个字段名。
```

示例：

```text
技术问题：参保地无法稳定匹配。

改成业务确认方可以直接判断的问题：
如果社保账单文件夹叫“唐人四川”，但花名册里参保地写的是“四川成都”，这两种名称是否代表同一个参保地？
如果是，请提供一张名称对应表。
```

### 5. 验收标准

必须提前告诉 Codex 怎么判断完成。

建议说法：

```text
完成后必须验证：
1. 单文件上传；
2. 文件夹上传；
3. zip 上传；
4. 不上传已有汇总表时，使用内置空模板新建；
5. 上传已有汇总表时，只追加或更新，不清空旧数据；
6. 输出格式和模板一致；
7. 日志内容清晰，业务使用者可以根据提示判断处理结果；
8. Mac 本地能跑，Windows 打包后也能跑。
```

## 三、和 Codex 协作的标准流程

不要一上来就让 Codex 写代码。正确流程分 7 步。

### 第一步：让 Codex 读需求和文件

提示词：

```text
先不要写代码。
请读取这个项目中的问题汇总表、相关附件和模板。
只分析需求X。
告诉我：
1. HR 要解决什么业务问题；
2. 输入文件是什么；
3. 输出文件是什么；
4. 当前资料是否足够实现；
5. 哪些地方需要问 HR 确认；
6. 如果实现，推荐怎么做。
```

这一步的目标不是写代码，而是防止 Codex 误解需求。

### 第二步：让 Codex 把问题改成业务确认方可回答的问题

提示词：

```text
把你认为需要确认的问题，改成业务确认方可以直接理解并回答的问题。
不要使用代码、字段映射、数据结构这些技术词。
每个问题都要具体到文件、表、字段或场景。
```

### 第三步：确认后再让 Codex 实现

提示词：

```text
HR 已确认，确认结果在这个文件中。
请结合原问题汇总表、备注、确认结果和模板，完善需求X。
要求：
1. 不要影响已完成需求；
2. 业务逻辑放到独立 tools 模块；
3. GUI 中增加对应入口；
4. 支持文件、文件夹、zip；
5. 输出默认放桌面独立文件夹；
6. 支持打开结果目录；
7. 增加简洁使用教程；
8. 完成后做真实数据验证。
```

### 第四步：要求 Codex 明确测试数据来源

这是非常关键的一步。

提示词：

```text
你测试用的数据源来自哪里？
请明确列出：
1. 哪些文件来自 HR 提供的真实模板；
2. 哪些文件是你复制出来的；
3. 哪些数据是构造的；
4. 为什么需要构造；
5. 最终输出路径是什么。
```

如果 Codex 使用了构造样例数据，必须让它说明这些样例是否能够代表真实模板。

### 第五步：让 Codex 做完整验证

提示词：

```text
对需求X做完整验证。
不要只跑一个主流程。
至少验证：
1. 单文件；
2. 多文件；
3. 文件夹；
4. zip；
5. 有已有汇总表；
6. 无已有汇总表；
7. 模板格式；
8. 日期；
9. 重复数据；
10. 新增行格式。
把验证结果和输出文件路径告诉我。
```

### 第六步：人工打开 Excel 看效果

自动测试只能验证数据处理逻辑，不能完全覆盖 Excel 的版式质量。

人工必须检查：

- 标题是否正确。
- 表头是否正确。
- 日期格式是否正确。
- 金额是否显示 `0`，不要显示 `-`。
- 新增行是否有边框。
- 新增行是否居中。
- 是否有残留测试数据。
- 是否有多余隐藏文件。
- 是否有业务使用者不需要关注的英文文件。
- 输出文件名是否清晰表达业务含义。

如果发现问题，直接截图给 Codex，并明确指出要改哪里。

示例：

```text
看截图，新增行没有边框，日期格式也和原数据不一样。
要求：
1. 插入的所有数据都居中；
2. 插入的所有数据都加边框；
3. 日期格式和原模板保持一致；
4. 不要影响其他需求。
```

### 第七步：让 Codex review 未提交代码

提示词：

```text
Please review my uncommitted changes。
重点检查：
1. 是否引入业务逻辑缺陷；
2. 是否误删模板；
3. 是否影响其他需求；
4. 是否有不该提交的本地文件；
5. 是否有打包发布风险。
```

review 通过后，再决定是否提交。

## 四、描述需求时要讲清楚的业务规则

很多 Excel 自动化失败，不是代码能力问题，而是业务规则没说清楚。

### 1. 输入范围

要说清楚：

```text
HR 可能上传：
- 单个 Excel；
- 多个 Excel；
- 一个文件夹；
- 一个 zip；
- zip 里还有文件夹；
- 文件夹里可能混有无关文件。
```

### 2. 是否追加到已有结果

要说清楚：

```text
支持两种场景：
1. 不上传已有汇总表：用内置空模板新建；
2. 上传已有汇总表：保留原数据，在原基础上追加或更新。
```

### 3. 唯一匹配规则

要说清楚用什么识别人。

优先级建议：

```text
身份证号优先。
如果没有身份证号，再用姓名。
姓名重复时不能只按姓名匹配，必须提醒。
```

错误说法：

```text
按员工匹配。
```

正确说法：

```text
新增、修改、去重时，优先按身份证号匹配。
如果身份证为空，再按姓名匹配。
如果同名不同身份证，不能合并，需要日志提醒。
```

### 4. 日期归属月份

要说清楚某条记录进入哪个月份。

例如异动表：

```text
增员：按入职日期归属月份。
减员：按离职日期归属月份。
转正：按转正日期归属月份。
调动：按调整日期归属月份。
如果文件名写 7 月，但入职日期是 6 月，最终进入 6 月汇总表。
```

### 5. 空值和默认值

要说清楚：

```text
没有工资显示 0，不要显示 -。
没有日期保持空白。
没有身份证号要跳过还是保留，需要日志提醒。
```

### 6. 输出格式

要说清楚：

```text
输出格式必须和模板一致。
新增行复制模板数据行格式。
新增数据居中。
新增数据加边框。
不要额外背景色。
不要生成业务使用者不需要关注的 json 文件。
```

## 五、给 Codex 的高质量提示词模板

### 模板1：需求分析

```text
请分析需求X，先不要写代码。

资料：
1. 问题汇总表：xxx.xlsx
2. 数据源文件夹：xxx/
3. 输出模板：xxx.xlsx
4. HR 补充确认：xxx.xlsx

请输出：
1. 需求目标；
2. 输入文件；
3. 输出文件；
4. 字段映射；
5. 业务规则；
6. 缺失信息；
7. 需要业务确认的问题；
8. 推荐实现方案。
```

### 模板2：实现一个需求

```text
开始实现需求X。

要求：
1. 使用现有项目架构；
2. 业务逻辑写在 hr_toolkit/tools/ 独立模块；
3. 公共能力放 common；
4. GUI 增加入口和使用教程；
5. 支持单文件、文件夹、zip；
6. 默认输出到桌面独立文件夹；
7. 支持打开结果目录；
8. 不影响已有需求；
9. 完成后用真实模板验证；
10. 验证结果告诉我。
```

### 模板3：指出界面问题

```text
看截图，这里不符合 HR 使用习惯。

问题：
1. xxx；
2. xxx。

要求：
1. 文案改成业务使用者可以直接理解的表达；
2. 危险操作必须二次确认；
3. 不要弹出需要用户自行判断技术选项的选择框；
4. 按钮要明确，例如“文件夹/压缩包”“文件”；
5. 不动业务逻辑。
```

### 模板4：指出 Excel 格式问题

```text
看截图，输出格式和模板不一致。

要求：
1. 新增行全部居中；
2. 新增行全部加边框；
3. 日期格式和原模板一致；
4. 标题和表头保持模板原样；
5. 不要清空已有数据；
6. 修改后重新验证。
```

### 模板5：检查是否有缺陷

```text
下面是 review 报告。
请逐条检查是否真实存在。
如果存在，在不影响现有业务的前提下修复。
如果是故意设计，说明原因。
如果不影响功能，可以暂不改。
修完后跑测试并告诉我结果。
```

### 模板6：发布和打包

```text
检查本次发布文件。
告诉我：
1. 哪些文件必须提交；
2. 哪些文件不需要进 Git；
3. Windows 打包后 Mac 需要复制哪个目录到服务器；
4. latest.json 指向哪个 zip；
5. 是否会影响自动更新。
```

## 六、后期修改和优化怎么做

### 1. 修改前先限定范围

不要说：

```text
当前视觉风格不符合交付要求，请重新调整。
```

要说：

```text
只改 GUI 视觉，不动业务逻辑、StringVar、函数签名、工具模块。
```

或者：

```text
只改需求6的日期格式和新增行边框，不影响需求1、2、3、4、5、7、8。
```

### 2. 每次只改一个问题

错误方式：

```text
同时修改界面、打包、需求6、需求7。
```

正确方式：

```text
先只修复需求6汇总表新增行没有边框的问题。
修完验证通过后，再处理花名册。
```

### 3. 保留截图和输出文件路径

给 Codex 的反馈越具体，修改越准。

建议格式：

```text
输出文件：
/Users/xxx/Desktop/异动汇总结果/xxx.xlsx

问题：
1. 减员表第 5 行日期变成 2026-04-30 00:00:00；
2. 原模板显示 2026/4/30；
3. 要保持和模板一致。
```

### 4. 要求 Codex 说明根因

出现问题时，不要只要求 Codex 直接修改。

建议说法：

```text
先告诉我根因，再修复。
要说明：
1. 为什么会出现这个问题；
2. 修改哪一段代码；
3. 是否影响历史输出；
4. 如何验证。
```

## 七、验收清单

每个需求完成后，至少按下面检查。

### 功能验收

- 单文件能处理。
- 多文件能处理。
- 文件夹能处理。
- zip 能处理。
- 混入无关文件不会导致程序异常退出。
- 不上传已有汇总表能新建。
- 上传已有汇总表能追加。
- 重复数据能去重或提醒。
- 缺字段时给出中文错误。
- 输出文件名清晰表达业务含义。

### Excel 验收

- 标题正确。
- sheet 名正确。
- 表头正确。
- 列宽合理。
- 边框完整。
- 日期格式正确。
- 金额格式正确。
- 新增行没有残留旧数据。
- 公式没有断。
- 不出现业务使用者不需要关注的临时文件。

### GUI 验收

- 每个工具入口清晰。
- 使用教程简短明确。
- 危险操作有二次确认。
- 日志先显示，再弹成功提示。
- 失败时日志能定位原因。
- 输出目录可打开。
- 默认输出到桌面独立文件夹。

### 发布验收

- Windows 可以打包。
- Mac 可以本地运行验证。
- Git 不提交本地附件和输出文件。
- 更新包只保留必要文件。
- `latest.json` 的版本号、下载地址、sha256 正确。
- 服务器能访问 zip。
- 旧版客户端能检测到新版本。

## 八、常见错误

### 错误1：让 Codex 自行假设模板

如果没有明确模板，Codex 可能会生成一个“数据正确但格式不对”的 Excel。

解决：

```text
必须指定最终模板文件。
要求新增行复制模板格式。
```

### 错误2：使用构造测试数据，但没有说明

构造测试数据可以用于补充边界场景，但不能替代真实模板验证。

解决：

```text
测试时必须列出数据来源。
真实模板必须跑一遍。
```

### 错误3：只看代码测试，不打开 Excel

Excel 工具必须人工打开看。

解决：

```text
要求 Codex 输出文件路径。
人工打开检查格式后，再反馈截图。
```

### 错误4：把本地附件提交到 Git

HR 原始数据通常不应该进仓库。

解决：

```text
让 Codex 检查 git status。
把附件、输出、缓存、临时文件加入 .gitignore。
```

### 错误5：一次要求改太多

改动范围越大，越容易引入缺陷。

解决：

```text
一个需求一个需求做。
一个缺陷一个缺陷修。
每次修完都验证。
```

## 九、推荐的完整工作流

```text
1. 收到 HR 需求、附件、模板
2. 让 Codex 只分析，不写代码
3. 让 Codex 整理 HR 确认问题
4. HR 补充确认
5. 让 Codex 实现一个需求
6. Codex 本地验证
7. 人工打开 Excel 检查
8. 截图反馈格式或业务问题
9. Codex 修复并复测
10. Review 未提交代码
11. 清理不该进 Git 的文件
12. Windows 打包
13. Mac 拉取发布文件
14. 上传服务器静态目录
15. 客户端检查更新
```

## 十、快速执行版

如果只记一件事，就记住这句话：

```text
不要只告诉 Codex “我要做一个工具”。
要同时给它：需求表、真实数据源、最终模板、HR 确认结果、验收标准。
```

推荐提示词：

```text
请先不要写代码。
先根据我提供的需求表、附件、模板，分析这个需求能不能实现。
如果能实现，告诉我输入、输出、业务规则、缺失问题和实现方案。
等我确认后，你再开始写代码。
实现后必须用真实模板验证，并告诉我输出文件路径。
```
